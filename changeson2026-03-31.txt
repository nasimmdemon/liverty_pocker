Liberty Poker - Completed Tasks
Date: 2026-03-31

1. Monitor Live Games — Firestore live feed no longer uses `status in ('waiting','playing')`. It now uses two parallel listeners (`status == 'waiting'` and `status == 'playing'`), merges documents by id, and sorts by last activity in the client. This avoids composite-index `failed-precondition` errors in production.
2. Monitor Live Games — The dashboard always shows at least three table rows. When fewer than three real games pass the staleness filter, the list is padded with synthetic “open” rows (sky “Open” badge, “Listing 1/2/3”, standard stakes copy). Nothing is labeled “reserved”; rows are monitor-only and are not written to Firestore.
3. Monitor stats — “Open Tables” uses the full displayed list length (live + padded). The waiting badge counts real waiting games plus open listings so an empty lobby still reads as three waiting lines when only synthetic rows show. “Tables with Bots” percentage uses live Firestore tables only.
4. Monitor Table Details — Card copy explains freshness windows and that at least three listings always appear; empty-state text adjusted while the padded list loads.
5. Live subscription errors — If the Firestore listener still fails, the UI shows the error message and three open listings so the layout stays consistent.

6. Monitor operations — Live tab: pause matchmaking bot table fill per tier/mode/sub-stake (Firestore `monitor_settings/controls`). Matchmaking reads this before creating a bot table after the wait timer; players are refunded with a clear toast if paused.

7. Monitor operations — Add open matchmaking table (Firebase-signed-in user as host, chosen pool from catalog) and Close table on each live row (marks `games/{id}` status `ended`). New helpers: `matchmakingPoolId.ts`, `matchmakingPoolCatalog.ts`, `monitorSettings.ts`, `closeGameRoomAsEnded`, `createMonitorMatchmakingOpenTable`.

8. Firestore rules — `monitor_settings/{docId}`: read allowed for everyone, write if `request.auth != null` (deploy rules for production).

9. Monitor demand scaling — Synthetic “Demand” rows per matchmaking pool (tier + Sit & Go vs Tournament + stake) from `max(queue size, matchmaking_flow events in last 60s)`; open option count = min(4, floor(pressure/6)) with pressure below 6 → 0 rows for that pool (then 6–11→1 … 24+→4). If no real games and no demand rows, three generic listings remain. “In Lobby / Waiting” includes queue players with Sit & Go vs Tournament breakdown.

10. Sit & Go lobby — Entrance amount slider and field are clamped to stakes-based bounds: minimum `(small blind + big blind) / 2 × 20`, maximum `(small blind + big blind) × 20` (e.g. 0.01/0.02 → $0.30–$0.60). Tournament buy-in range is unchanged (still tied to balance).

11. Public lobby header — Removed the centered Public pill from the top bar (funds stay prominent). “Public” is indicated in the matchmaking section as “🌐 Public — matchmaking”; private flow has “🔒 Private tables” above create/join. Header **FUNDS** shows **wallet minus the selected buy-in** (live as you move the slider); after `deductFunds` for matchmaking, a small held offset avoids double-subtracting until the flow ends.

12. Public lobby tier UX — Choosing a stake closes the expanded tier panel and shows a pill-style **SELECTED: … (SB/BB)** row; `matchmakingTier` keeps tier + stakes for join while the panel is collapsed; tier cards stay highlighted for the committed tier.

13. Matchmaking fairness — Pool docs store per-user public match counts, pair blocks (no rematch until each player has played **three** more recorded matches after sharing a table), and a **25s** re-pair cooldown after a game ends. Queue pairing and open-table joins respect blocks + cooldown; `recordMatchmakingGameEnded` runs when a matchmaking game reaches **ended** (idempotent per game). Tests cover pairing helpers.

14. Monitor (Live tab) — Extended operations: edit **matchmaking await time** and **player re-pair cooldown** (stored on `monitor_settings/controls`, read by clients). **Reserve** block: total max plus per-tier max / allocated / current (planning fields; even split ÷4 shown). **Tier table**: live flow pressure and open stake counts (Sit & Go vs Tournament), human-only badge when all pools in a tier have bot fill paused, **Pause tier** / **Enable tier** for all stakes. **Economy ledger** (`monitor_settings/economy_summary`): video payouts, wins vs bots, losses vs bots, expired funds — auto-increment from Watch & Earn, solo public exits (bot tables), and multiplayer vs bots; operators can edit totals. Matchmaking skips the bot timer when a tier is fully paused (infinite human wait); cooldown duration follows monitor setting.

15. Monitor glossary — Open card explains operator semantics: **withdraw** = reserve shrank; **DB record size ↑** = bot/house lost (to player or player vs player); **expiry** = pre-withdrawn. **Reserve filled (Current $)** differs by tier: **Human** = bot subsidies + free player credits combined; **Dog / Rat / Cat** = money bots received to play vs people. Per-tier notes appear in the reserve table; economy card cross-links withdraw wording.
